# AI Cyber Risk Prioritisation Engine

A RAG-based system that ranks vulnerabilities by real world risk (not just CVSS), then retrieves official **NIST 800-53** remediation guidance for each top risk and explains it in plain English using an LLM.

Live demo: https://cyberriskprioritizer-f7pyvfdpnnjqxcvhfjvtu8.streamlit.app

---

## What it does

Given a dataset of assets, vulnerabilities, threat intelligence, and business services, the system:

1. **Enriches** each vulnerability by joining it to its asset's context, business criticality, and active threat campaigns.
2. **Scores** every vulnerability using a weighted model based on the factors the threat intelligence team prioritised: internet exposure, active exploitation, ransomware association, business criticality, and missing controls (e.g. no EDR).
3. **Retrieves** the most relevant NIST 800-53 control for each top risk using semantic search (embeddings + cosine similarity) over the real NIST catalogue — not the LLM's memory.
4. **Explains** each risk in a plain-English brief generated by an LLM, grounded strictly in the scored facts and retrieved NIST guidance.

## Risk scoring model

Each vulnerability is scored out of ~100:


 CVSS severity  score × 2 (0–20) 
 Internet-facing  +20 
 Actively exploited  +20 
 Ransomware-linked  +15 
 Business criticality | Critical +15 / High +10 / Medium +5 |
 Missing EDR  +10 

The weighting reflects the priority order recommended by the threat intelligence report: exposure and active exploitation drive the biggest score changes, while business criticality acts as a tie-breaker between equally exploitable risks.

## Tech stack

- **Python / pandas** — data joining, enrichment, and scoring
- **sentence-transformers (all-MiniLM-L6-v2)** — embeddings for RAG
- **NIST SP 800-53 Rev.5** — remediation source (retrieved, not hardcoded)
- **Groq (Llama 3.3 70B)** — LLM for plain-English briefs
- **Streamlit** — web app + deployment

## Supporting questions

### 1. How did you decide what was structured vs unstructured data, and how did that change your approach?

The assets, vulnerabilities, threat intelligence, and business services were structured CSVs — clean rows and columns with exact values — so I basically queried and joined them directly using pandas (matching on shared keys like `asset_id` and `cve`). The NIST 800-53 catalogue is unstructured , where the right control might use completely different wording from a given risk, so exact matching wouldn't work. For that I embedded each control into vectors and used semantic similarity to retrieve the closest one. In short: structured data gets queried, unstructured data gets embedded and retrieved — different data shapes need different tools.

### 2. Where does your system fall short, and what would you do with more time?

A few honest limitations:
- **Query phrasing affects RAG quality.** I found that metadata-heavy risk descriptions drifted toward generic governance controls (PL-3), while descriptions focused on the security problem matched the correct control (SI-2). With more time I'd tune the query construction per vulnerability type.
- **The top 5 can cluster on the same assets.** Because two VPN appliances each had multiple critical flaws, 4 of my top 5 were the same two machines. I'd add an option to roll up to asset-level so the top 5 covers distinct problems.
- **Small embedding model.** all-MiniLM-L6-v2 is fast but limited; a larger model would give sharper matches according to my research.
- I de-duplicated threat-match rows (some CVEs matched multiple campaigns, inflating the count from 114 to 117) before scoring, so flaws weren't double-counted.

### 3. How would this work in the real world / at scale?

In production I'd replace the static CSVs with live feeds (a real vulnerability scanner, a CISA KEV auto-refresh, a threat intelligence platform), pre-compute and cache the NIST embeddings rather than rebuilding them each run, and keep a human analyst in the loop to review the AI's output rather than trusting it blindly. The core architecture — score by context, retrieve grounded guidance, explain — would stay the same, but the data sources would be automated and continuously updated.
